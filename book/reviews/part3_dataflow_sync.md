评审范围：

- `book/chapters/ch06_data_buffer.md`（Data Buffer——隐式的桥梁，约 380 行）
- `book/chapters/ch07_weight_sync.md`（weight sync——四条传输路径，约 410 行）

整体定调：两章质量在全书中属于上游水准。论点鲜明、章内结构干净、
Apply This 与"下一站"格式对称。**最值得动手修的是两件事**：
(1) Ch 6 的 6.5 节"Sample 生命周期"与第 5 章 Sample 状态机重叠

## 2. 流畅性

### Ch 6：契约 vs 模块论点是否被稀释

整章 5 节，5 节都能挂回"契约中心 + 实现下沉"主论点：6.1 讲契约、
6.2 讲三层实现、6.3 讲过滤逻辑外置、6.4 讲 checkpoint 不入、
6.5 讲 Status 与方法引用。结构干净。

**问题在 6.5 节本身**：它包含两件事——Sample 状态机示意图 + 方
法引用切分读写权限的"深入剖析"。前者与第 5 章 Sample 状态机
（5.2 节）显著重叠（第 5 章已经讲过 PENDING / COMPLETED /
TRUNCATED / ABORTED / FAILED 五态）；本章重复一次状态机图，只
为引出 `ABORTED → Buffer` 那一条边。**建议**：删掉完整状态图，
只保留单独那条"ABORTED → Buffer"的小流程图（3 个节点），把
节省的篇幅留给方法引用切权限那段——那才是 6.5 的独到贡献。

### Ch 7：5 个小节的节奏

5 节分布：carrier 选择（38-89 行，~50 行）→ 共享层（90-155 行，
~65 行）→ Delta（156-216 行，~60 行）→ Disk（217-242 行，
~25 行）→ FP8 vs int4（243-319 行，~75 行）。**Disk 节明显
轻**——只解释了 disk 解锁异构 GPU 一件事，可以与 7.1 的部署矩
阵表合并讲完，单独成节略尴尬。**Delta 节与 FP8 节都偏重**——但
都言之有物，不建议进一步切分。

**叙事松脱的位置在 7.2**：标题是"共享层"但实际讲了三件事——
megatron_to_hf 中间层、10 个 converter 的差异表、注册机制是 if-
elif。读者会困惑这一节到底要论证什么。**建议**：把 7.2 的论点
明确成"为什么共享中间层 + carrier 分叉，而不是反过来或者全部
抽象"，把现在的三件事都收束到这一句论点下。

### 重复内容

Ch 6 的 6.1 节里"`add_samples` 默认抛错"被讲了 2 次（第 47 行
代码里、第 71-74 行散文里），第二次可以压缩为"如前所述"或
直接删除散文中的复述。

## 4. 缺失内容

### Ch 6「权限按层切开」——讲清楚了

6.5 末尾的深入剖析框（270-292 行）把 `data_source.get_samples`
方法引用 vs `data_source` 完整对象的差异讲得很到位，配上对应
代码片段和"权限通过函数签名表达"的提炼。**唯一缺的**是一句话
说明这种做法的"反面"——如果不切，会有什么具体的 bug？建议
补一句："否则内层生成循环可能在 abort 时直接调
`data_source.add_samples`，绕过外层 `generate_rollout` 控制写
入时机的责任。"——让"权限切分"的好处具体化。

### Ch 7「FP8 vs int4 不对称」——讲清楚了，但结构可优化

7.5 节论证逻辑完整：FP8 send-side 的两个理由（省带宽 + ue8m0
比特一致）+ int4 receive-side 的理由（packing 与 GEMM kernel
强耦合）。**问题不在内容缺失，在排版**：

- "第一反应"那一段（第 248 行）属于读者心理模型对比，但下文用了
  "slime 不完全是这样"接续，再下面又用"两种量化方案走的是不同
  方向"再开一个并列——三层"驳论"结构让读者跟丢。**建议**：删掉
  "第一反应 → slime 不完全是这样"那两段，直接以"FP8 在发送前
  做，int4 在接收后做"作为本节论点开门见山。
- 最后一段（281-285 行）的总结"这两个量化方案的分叉不是设计不
  一致，是被各自的成熟度决定的"——这句话是本节最重要的洞见，
  应该作为节标题或开篇句重复一次，而不是埋在末尾。

### 其他缺口

- Ch 6 没有讲清"为什么默认 buffer 是 Python list 而不是
  `collections.deque` 或某种 priority queue"——读者可能会问。
  **可在 6.2 末尾补一句**："默认实现选 Python list 不是性能选
  择，是承认 'buffer 的选择策略 = buffer_filter 注入函数' 这条
  设计：容器本身只需支持 append / pop / len，再复杂的语义都从
  外部 hook 表达。"
- Ch 7 缺一个"端到端时间线"的视角——读者读完 5 节知道每条路径
  长什么样，但不知道"一次完整的 update_weights 在墙钟上要多长
  时间"。**可在 7.1 或 7.4 末尾补一段散文**：给出"NCCL 全量：
  数十秒级；disk 全量：分钟级；delta：百 MB/s 链路上仍是秒级"
  这种粗略量级（不要给精确数字），让"四条路径不是性能档位"那个
  论点不被误解为"性能完全无关"。

## 5. 需要的图示

按优先级排：

**P0：Ch 7 替换 7.3 的 ASCII pipeline（189-203 行）为
sequenceDiagram**。当前的 ASCII 用空格缩进模拟管线，读者无法
看出"GPU compute / 侧 stream H2D / 侧 stream D2H"三者在时间
轴上如何重叠。建议：

```
participant: GPU compute / d2h_stream / h2d_stream / NCCL
横轴：chunk_k-1 / chunk_k / chunk_k+1
事件：chunk_k diff+encode（GPU）与 chunk_k-1 D2H、chunk_k+1 H2D 并行
```

这张图本身就解释了"为什么 snapshot 要在 pinned CPU memory"。

**P1：Ch 7 在 7.1 加一张"部署形态 → 路径"决策树（flowchart）**。
当前 7.1 用 4 行 if-else 伪代码 + 一张表表达决策——伪代码是从代
码视角，表是从形态视角，但读者真正需要的是"我的场景是 X，应该
选哪条路径"的决策图。三个 yes/no 问题（colocate？delta？disk
transport？）一张 flowchart 解决。

**P2：Ch 6 在 6.2 末尾把现有 graph LR（103-121 行）改成"三层
对应三种 RL 拓扑"的合并图**。现在的图只展示了静态结构（契约层
→ 默认实现层 → 外置层），紧跟一个独立的"三种拓扑"表格。把两个
合并成一张图，每个实现节点旁标注它对应的 RL 拓扑（同步 / fully
async / agent long-tail），一张图把现在分散的两件事讲完。

**P3：Ch 6 用单独的 sequenceDiagram 画"一次 partial rollout
中断 + 回流"的时序**。现在 6.5 节的状态机图（244-255 行）画的
是 sample 的"位置"，但 partial rollout 的关键是"时序"——
get → generate → abort → add_samples 回流 → 下一轮再 get 时
buffer 优先。这个时序更能讲清楚"为什么 Buffer 状态在 ckpt 里
被丢弃也无大碍"。

**P4（可选）：Ch 7 7.5 节加一张"FP8 vs int4 路径分叉"图**。
两条路径并列：FP8 路径在发送前 quantize → wire 是 FP8 + scale →
recv 端无操作；int4 路径在发送前 bf16 → wire 是 bf16 → recv 端
restore + post_process_quantization 两阶段 hook。图能让"对称
路径在哪一步分叉"一目了然。

## 6. 跨章一致性

### 两章风格对称性

非常好。两章都以"一个反差点立住论点 → 5 节展开 → Apply This
5 条 → 下一站预告"的结构。Apply This 都是 5 条，每条都是
"模式名 + 怎么改造 + 陷阱"三段格式，与 Ch 4 / Ch 5 一致。

### 术语一致性

- "鸭子类型契约"（Ch 7 第 61 行）vs "契约"（Ch 6 全章）——同义但
  Ch 6 完全没用"鸭子类型"这个词，Ch 7 突然引入，建议 Ch 7 第一
  次用时加一句"和 Ch 6 那个 5 方法接口同样是契约，只是更隐式——
  不通过 abc.ABC 显式声明，只靠调用方按形状用"。
- "carrier" 一词在 Ch 7 内部使用一致，但首次出现（第 38 行 7.1
  节标题）没有定义。**建议**在 7.1 开篇加一句："carrier 在这一
  章指负责把权重从训练侧传到推理侧的具体类——四个 carrier 对应
  四条路径。"

### 矛盾点

未发现事实性矛盾。一处轻微不一致：

- Ch 6 第 20 行说契约"被一个 26 行的抽象基类（`DataSource`）固
  定住"；Ch 6 第 53 行又说"整个接口**只有 5 个方法**——`get_samples
  / add_samples / save / load / __len__`"。"26 行"是文件行数还
  是接口大小？读者会困惑。**建议**统一表述为"5 个方法 / 26 行
  代码（含 docstring）"。
- Ch 7 第 60-62 行说"四个 carrier 类**没有公共基类**"，但 7.2
  节标题"共享层"会让读者预期"有共享代码"。**建议**：在 7.1 末
  尾或 7.2 开头加一句过渡，明确"没有公共基类不等于没有共享代
  码——共享发生在中间层（HF 转换），而不是顶层 carrier 抽象。"

### Apply This 节格式

两章 5 条数量一致，三段（模式 / 适配 / 陷阱）格式一致。**Ch 6
的 #5（方法引用切权限）与 6.5 节的深入剖析框 high overlap**——
深入剖析框已经讲了一遍，Apply This #5 又讲一遍。建议把 Apply
This #5 压缩到 2-3 句，主要保留"陷阱"那段（GC 与生命周期问题
是深入剖析框没讲的）。
